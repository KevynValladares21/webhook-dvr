# Webhook DVR

Webhook DVR is a lightweight webhook capture and replay service built with FastAPI.

It records incoming webhook HTTP requests (body, headers, query parameters) and allows developers to inspect and replay them later. This helps debug third-party integrations such as Stripe, GitHub, Shopify, and similar webhook-based systems.

---

## Motivation

Webhook integrations are often difficult to debug because:

- Events fire asynchronously and unpredictably
- Payloads are lost if a server is down or fails
- Reproducing failures can be difficult
- Replaying historical events is rarely supported

Webhook DVR acts as a **"DVR for webhooks"**, recording events for inspection and replay.

---

## Current Features

- Capture webhook requests via ingestion endpoint
- Store:
  - Raw request body
  - Headers
  - Query parameters
  - Source label
  - Timestamp
- Persist events in PostgreSQL
- Retrieve recent events
- Retrieve events by ID
- Replay stored events to a configured target URL
- API key authentication for protected endpoints
- Local PostgreSQL setup with Docker Compose
- Test receiver for local replay testing

## Getting Started

### Clone the repository

```bash
git clone <repo-url>
cd webhook-dvr
```

### Create virtual environment

```bash
python -m venv .venv
source .venv/bin/activate
```

### Install dependencies

```bash
pip install -e ".[dev]"
```

### Configure environment

Create a `.env` file:

```bash
cp .env.example .env
```

Example values:

```env
ENV=dev
API_KEY=dev-secret-key
REPLAY_TARGET_URL=http://localhost:9000/test
DATABASE_URL=postgresql+psycopg://webhook:webhook@localhost:5432/webhook
```

### Start PostgreSQL

```bash
docker compose up -d
```

### Create database tables

If the database is fresh, create the tables:

```bash
python - << 'EOF'
from app.db.models import Base
from app.db.session import engine

Base.metadata.create_all(engine)
EOF
```

### Run the server

```bash
uvicorn app.main:app --reload
```

API documentation will be available at:

```text
http://localhost:8000/docs
```

## Usage

Protected endpoints require:

```text
X-API-Key: dev-secret-key
```

### Ingest a webhook

```bash
curl -X POST "http://localhost:8000/ingest/test" \
  -H "X-API-Key: dev-secret-key" \
  -H "X-Custom-Header: hello" \
  -H "Content-Type: application/json" \
  -d '{"hello": "world"}'
```

Response:

```json
{
  "event_id": "..."
}
```

---

### List recent events

```bash
curl -H "X-API-Key: dev-secret-key" \
  http://localhost:8000/events
```

---

### Retrieve an event

```bash
curl -H "X-API-Key: dev-secret-key" \
  http://localhost:8000/events/<event_id>
```

---

### Replay an event

Start the local test receiver in another terminal:

```bash
uvicorn test_receiver:app --port 9000
```

Then replay a stored event:

```bash
curl -X POST "http://localhost:8000/events/<event_id>/replay" \
  -H "X-API-Key: dev-secret-key"
```

If replay succeeds, the test receiver terminal should print the received body and headers.

## Design Notes

Webhook DVR intentionally stores raw request data to preserve fidelity for debugging and replay. Webhook payloads are not assumed to be JSON, since real-world webhook providers may send different content types.

Replay is synchronous in the current MVP to keep the system simple and easy to reason about. In a production system, replay would likely be handled by a background worker with retries and replay attempt tracking.

Some headers, such as `host` and `content-length`, are filtered before replay because they are connection-specific and should be regenerated by the HTTP client.

This project emphasizes:

- Backend architecture clarity
- Event-driven system thinking
- Reliability and debuggability of integrations
- Simple production-oriented patterns such as configuration, persistence, and authentication
