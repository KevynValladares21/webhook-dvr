# Webhook DVR

Webhook DVR is a small FastAPI backend service for capturing, inspecting, and replaying webhook requests.

It acts like a DVR for HTTP webhooks: incoming requests are stored in Postgres and can later be inspected or replayed to another service for debugging.

## Why This Exists

Webhook integrations can be difficult to debug because events are often one-shot, asynchronous, and hard to reproduce. If a receiving service is down, misconfigured, or fails to process a payload correctly, the original request may be difficult to recover.

Webhook DVR provides a simple way to capture incoming webhook traffic, inspect stored events, and replay them to another service during development or debugging.

## Features

- Capture incoming webhook requests
- Store request body, headers, query parameters, source, and timestamp
- List recent webhook events
- Inspect a single stored event
- Replay a stored event to a configured target URL
- API key authentication
- Postgres persistence
- Docker Compose local database setup

## Architecture

```text
External Service
      |
      | POST /ingest/{source}
      v
FastAPI API
      |
      | stores event
      v
Postgres
      |
      | fetch event
      v
Replay Endpoint
      |
      | sends stored body + selected headers
      v
Target Service
```

## Data Model

Each stored webhook event contains:

- `id`
- `source`
- `body`
- `headers`
- `query_params`
- `received_at`

## API Endpoints

### Health Check

```http
GET /healthz
```

### Ingest Webhook

```http
POST /ingest/{source}
X-API-Key: dev-secret-key
```

Stores the incoming request and returns an event ID.

### List Events

```http
GET /events
X-API-Key: dev-secret-key
```

Returns the 10 most recent webhook events.

### Get Event

```http
GET /events/{event_id}
X-API-Key: dev-secret-key
```

Returns the full stored event.

### Replay Event

```http
POST /events/{event_id}/replay
X-API-Key: dev-secret-key
```

Replays the stored event to the configured `REPLAY_TARGET_URL`.

## Local Development

### 1. Create virtual environment

```bash
python -m venv .venv
source .venv/bin/activate
```

### 2. Install dependencies

```bash
pip install -e ".[dev]"
```

If you are not installing dev tools, use:

```bash
pip install -e .
```

### 3. Configure environment

```bash
cp .env.example .env
```

Your `.env` should contain:

```env
ENV=dev
API_KEY=dev-secret-key
REPLAY_TARGET_URL=http://localhost:9000/test
DATABASE_URL=postgresql+psycopg://webhook:webhook@localhost:5432/webhook
```

### 4. Start Postgres

```bash
docker compose up -d
```

### 5. Start the API

```bash
uvicorn app.main:app --reload
```

### 6. Start the test replay receiver

In another terminal:

```bash
uvicorn test_receiver:app --port 9000
```

## Demo

### Ingest a webhook

```bash
curl -X POST "http://localhost:8000/ingest/test" \
  -H "X-API-Key: dev-secret-key" \
  -H "X-Custom-Header: hello" \
  -H "Content-Type: application/json" \
  -d '{"hello": "world"}'
```

Example response:

```json
{
  "event_id": "your-event-id"
}
```

### List events

```bash
curl -H "X-API-Key: dev-secret-key" http://localhost:8000/events
```

### Inspect one event

```bash
curl -H "X-API-Key: dev-secret-key" \
  http://localhost:8000/events/your-event-id
```

### Replay event

```bash
curl -X POST "http://localhost:8000/events/your-event-id/replay" \
  -H "X-API-Key: dev-secret-key"
```

If replay works, the `test_receiver` terminal should print the received request body and headers.

## Design Decisions

### Synchronous Replay

Replay is currently synchronous to keep the MVP simple and easy to reason about. A production version could move replay execution to a background worker backed by a queue.

### Raw Request Capture

The service stores the raw request body instead of assuming JSON because real webhooks may use different content types. This preserves request fidelity for debugging and replay.

### Header Filtering During Replay

Some headers, such as `host` and `content-length`, are connection-specific and should be regenerated by the HTTP client. The replay logic filters these headers before forwarding the request.

### API Key Authentication

The MVP uses simple API key authentication to protect ingestion, listing, inspection, and replay endpoints.

### Environment-Based Configuration

Database connection strings, replay targets, and API keys are configured through environment variables rather than hardcoded in route logic.
